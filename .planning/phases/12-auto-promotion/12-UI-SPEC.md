# Phase 12 — UI Design Contract

> Visual and interaction contract for replacing `PromoteTimelinePlaceholder.tsx` with a populated timeline driven by the new `pg_promote_event` WS stream and `GET /api/pg/discovery/promote-events` REST seed. Generated by gsd-ui-researcher.

---

## Scope

This contract covers exactly **one frontend surface change**: the empty Phase 11 `PromoteTimelinePlaceholder` (web/src/components/Discovery/PromoteTimelinePlaceholder.tsx, 26 lines) is replaced in place by a populated `PromoteTimeline` component. The component subscribes to a new WS event `pg_promote_event` (CONTEXT D-11), seeds from `GET /api/pg/discovery/promote-events` (CONTEXT D-12) returning at most 1000 events from `pg:promote:events` LRANGE 0 -1 (CONTEXT D-10), and renders newest-first.

**In scope:** timeline item visual contract, populated/empty/loading/error states, WS append + reconnect re-seed dedupe, item-count cap, accessibility semantics, EN+zh-TW i18n keys, copywriting.

**NOT in scope (deferred):** entire PriceGap tab restructure (Phase 16 PG-OPS-09), per-fill Telegram alerts (PG-LIVE-04, v2.3+), score-vs-realized-fill calibration view (PG-DISC-05, v2.3+), separate top-level Discovery tab (Phase 16). The component file path, parent (`DiscoverySection.tsx` line 133), surrounding card layout, and the `pricegap.discovery.timeline.title` heading are inherited from Phase 11 and MUST NOT change.

### States

| State | Trigger | Render |
|-------|---------|--------|
| Loading | `promoteEvents == null && promoteEventsLoading == true` (initial mount, seed in flight) | Card heading + `text-xs text-gray-500` body with `pricegap.discovery.timeline.loading` copy |
| Empty | `promoteEvents != null && promoteEvents.length === 0` (seed succeeded, no events yet) | Card heading + `text-xs text-gray-500` body with `pricegap.discovery.timeline.empty` copy |
| Populated | `promoteEvents.length > 0` | Card heading + scrollable list with up to MAX_RENDERED items (see cap rules) |

### Item count cap

- **MAX_RENDERED = 50** items by default in the visible list.
- If the seed returns more than 50, render exactly 50 (the newest 50) and append a footer link `pricegap.discovery.timeline.loadMore` ("Show all (N)" — count interpolated). Click toggles `showAll` boolean state which renders all events received (still capped at 1000 by D-10 LTrim).
- **Hard cap = 1000** items in client memory. The hook trims the in-memory array to length ≤1000 on every WS append (slice last 1000), mirroring the server LTrim and the existing `usePgDiscovery` score-trim pattern at line 206.

## Interaction Contract

### Initial mount (REST seed)
1. On mount, hook calls `GET /api/pg/discovery/promote-events` with `Authorization: Bearer ${arb_token}` (matching Phase 11 hook pattern at lines 88-96).
2. Response is the LRANGE 0 -1 of `pg:promote:events`. **Server returns oldest-first by Redis LIST semantics** — the hook reverses to newest-first before storing.
3. Loading state renders until response settles (success or error).

### Live append (WS push)
1. WS event `pg_promote_event` (D-11) handled by extending the `usePgDiscovery.ts` switch at line 184. Add a `case 'pg_promote_event'` that prepends the event to `promoteEvents` (newest-first ordering).
2. **Dedupe key:** composite `(ts, action, symbol, long_exch, short_exch, direction)`. If an arriving event matches the first item by all six fields, drop it (covers the late-seed-vs-WS-stream race described in CONTEXT specifics).
3. **Cap:** after prepend, slice to `length ≤ 1000`.

### WS reconnect re-seed (race coverage)
1. When `wsConnected` transitions `true → false → true`, the hook re-issues the REST seed and **replaces** the in-memory list with the seeded result. This recovers any events missed during the disconnect window.
2. The dedupe-on-prepend rule (above) prevents WS messages that arrive in flight during re-seed from creating duplicates.

### Newest-first ordering invariant
- Visible list ordering: index 0 = newest event. Append direction = top.
- Sort key on REST seed: descending `ts` (unix ms). For exact ts ties (server-side same-ms emit), tiebreak by Redis insertion order (server LIST is FIFO, oldest at index 0 — reversal preserves intent).

### Click / hover behavior — v1 decision
- **No click-to-detail in v1.** Rows are visually informative but not interactive. RESEARCH does not call for a detail drawer, and Phase 12's Telegram message already carries the full payload an operator needs. Adding a drawer adds surface area without a clear user job in this phase.
- **Hover affordance:** `hover:bg-gray-800/50` only — purely visual depth; no cursor change, no tooltip required.
- **Future-proofing:** the row container should accept an optional `onClick?: (event: PromoteEvent) => void` prop on the per-row sub-component but the parent does NOT pass one in v1. This lets Phase 16 add detail-drawer behavior without restructuring.

### Keyboard navigation
- The scrollable list (`max-h-[480px] overflow-y-auto`) is naturally keyboard scrollable when focused — `tabindex="0"` on the scroll container + `role="list"` + each row `role="listitem"`.
- Up/Down arrow keys: native scroll behavior (browser default once the list has focus).
- No row is individually focusable in v1 (rows are not interactive). Adding row focus + Enter activation is deferred until click-to-detail ships.

## Accessibility Contract

### aria-live semantics
- **The scrollable list container is `aria-live="polite"`** with `aria-atomic="false"`. Polite (not assertive) because promote/demote events are informational; the operator does not need to interrupt their current task to acknowledge them — Telegram alerts (D-13) already provide the urgent path.
- The `role="list"` is set on the scroll container. Each row is `role="listitem"`.

### Screen-reader announcement format
When a new event is prepended, the announce-string shape is:
- Promote: `{i18n: pricegap.discovery.timeline.sr.promote} {symbol} {long_exch} {short_exch} score {N}`
- Demote: `{i18n: pricegap.discovery.timeline.sr.demote} {symbol} {long_exch} {short_exch} score {N}`

Implementation: render an off-screen `<span class="sr-only">` inside each row containing the announce string; aria-live picks it up automatically. Visible row text remains compact.

### Color contrast
- Action chip foreground vs background: `#0ecb81` on `bg-[#0ecb81]/15` over `bg-gray-900` and `#f6465d` on `bg-[#f6465d]/15` over `bg-gray-900` — both are reused **as-is** from the existing `discovery-status-pill` (DiscoverySection.tsx:60) and `discovery-banner-errored` (DiscoveryBanner.tsx:65), which already passed Phase 11 review. No new color contrast work required.
- Body text `text-gray-300` on `bg-gray-900` and metadata `text-gray-500` on `bg-gray-900` are also inherited.

### Focus visibility
- The scroll container has `focus-visible:outline-none focus-visible:ring-2 focus-visible:ring-gray-700` — keyboard users see a clear ring without shifting layout.

### `prefers-reduced-motion`
- No animations are added in v1. The chip is static; new rows appear without animation. Honors `prefers-reduced-motion` by default (no opt-out work needed).
