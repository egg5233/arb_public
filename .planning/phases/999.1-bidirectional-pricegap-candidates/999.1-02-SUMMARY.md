# Phase 999.1 Plan 02: Validator Direction-Field Branch Summary

Direction enum validation lands in `validatePriceGapCandidates` with a paired post-validation normalization loop in the POST /api/config apply path — backward-compat default of "pinned" applied silently, unknown values rejected with HTTP 400 echoing the offending symbol and value.

## What Was Built

One validator branch added (5 lines), one normalization loop added in the caller (3 lines), six unit tests covering the full enum surface (105 lines). The validator now accepts `""`, `"pinned"`, `"bidirectional"`; everything else is rejected with `candidate[%d] %s: direction must be "pinned" or "bidirectional" (got %q)`. The handler apply block in `internal/api/handlers.go` invokes `models.NormalizeDirection` on each candidate after validation succeeds and before the active-position guard runs, so persistence and reload always see the canonical "pinned" value rather than empty string. Pitfall 3 mitigation honored: case-sensitive enum, no auto-lowercase, no silent acceptance.
