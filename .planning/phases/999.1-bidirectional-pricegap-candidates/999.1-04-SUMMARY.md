# Phase 999.1 Plan 04: Direction Modal Toggle Summary

Added a two-option Direction radio toggle (Pinned / Bidirectional) to the existing Add/Edit candidate modal in `web/src/pages/PriceGap.tsx`. Defaults to "pinned" for new candidates and any legacy candidate without the field; Save handler now includes `direction` in the POST /api/config payload. Plan 03 i18n keys consumed unchanged.

## What Was Built

- **TS type extension** — `PriceGapCandidate` interface gains `direction?: 'pinned' | 'bidirectional'` (matches the JSON tag from Plan 01's Go struct).
- **Form state** — new `formDirection` useState typed `'pinned' | 'bidirectional'`, defaulting to 'pinned'.
- **Edit pre-populate** — `openEditor` now sets `setFormDirection(target.direction ?? 'pinned')` so candidates persisted before Plan 01 (no `direction` field) display as Pinned.
- **Add default** — `openEditor` Add branch resets to 'pinned'.
- **Local validation** — `validateLocalForm` rejects unknown direction with the i18n key `pricegap.candidates.modal.direction.validation.invalid` (defense-in-depth — Plan 02 server validator catches the same case).
- **Save payload** — `handleEditorSave` writes `direction: formDirection` into the draft candidate sent to POST /api/config.
- **Modal UI** — `<fieldset>` with two `<input type="radio" name="pg-direction">` controls, an inline help paragraph, and inline error text. Tailwind classes mirror existing modal labels/inputs (gray-300 label, gray-500 help, red-400 error).
