# Phase 999.1 Plan 05: Bidirectional Pricegap — Regression + E2E Coverage Summary

End-to-end paper-mode coverage for bidirectional inverse fire + backward-compat regressions for legacy candidates and legacy position JSON + i18n parity for direction.* keys.

## What Was Built

Two test files, eight tests, zero production-code changes. The Go integration test drives the full tracker pipeline (detector → 4-bar persistence gate → runTick → risk gate → openPair → store) on a sustained negative spread and asserts wire-side leg roles are swapped while the configured tuple is preserved. A backward-compat companion test asserts that the same setup with a legacy `Direction=""` candidate produces zero positions — pinning the Plan 01 latent-bug fix. A third test hand-crafts pre-Phase-999.1 JSON in miniredis and confirms it decodes with zero-value defaults on the new fields. The frontend parity test asserts identical key sets between `en.ts` and `zh-TW.ts` with explicit coverage of the 5-key `pricegap.candidates.modal.direction.*` namespace.

## Tasks Completed

### Task 1: Backend E2E paper-mode test for bidirectional inverse fire

Created `internal/pricegaptrader/integration_test.go` with three tests:

- **TestE2E_BidirectionalInverseFire_PaperMode** — primes 4 bars of -250 bps spread on a `Direction=bidirectional` candidate, runs `runTick` in paper mode, asserts: (a) zero `PlaceOrder` calls on either leg, (b) wire-side `LongExchange=bybit` / `ShortExchange=binance` (swapped from configured `binance`/`bybit`), (c) `CandidateLongExch=binance` / `CandidateShortExch=bybit` (configured tuple preserved for Phase 10 D-11 guard), (d) `FiredDirection=inverse`, (e) `Mode=paper`, (f) `Status=open`.
- **TestE2E_PinnedRejectsInverseSign_BackwardCompat** — same setup but `Direction=""` (legacy). Asserts zero placed orders and zero active positions — pinned mode must reject inverse spread (Plan 01 latent-bug fix).
- **TestE2E_LegacyPositionDecode_BackwardCompat** — hand-crafts a pre-Phase-999.1 JSON payload via `miniredis.HSet` (no `FiredDirection` / `CandidateLongExch` / `CandidateShortExch` fields), then loads via `db.GetActivePriceGapPositions()`. Asserts new fields default to zero values and legacy fields decode normally.

Reuses Phase 7 e2e harness (`newE2EClient`, `e2eCandidate`, `e2eCfg`, `newStubExchange`). Added one helper: `feedFiringSpreadInverse` (sign-flipped mirror of `feedFiringSpread`).

### Task 2: Frontend i18n parity test

Created `web/src/i18n/parity.test.ts` using Node 22 native test runner with `--experimental-strip-types`. Five tests:

- Total key-count parity between `en.ts` and `zh-TW.ts`.
- Every `en` key exists in `zh-TW`.
- Every `zh-TW` key exists in `en`.
- The `pricegap.candidates.modal.direction.*` namespace has exactly 5 keys in both locales.
- All 5 required direction keys (`label`, `option.pinned`, `option.bidirectional`, `help`, `validation.invalid`) are present with non-empty values in both locales.

Per Plan §Step B fallback, `PriceGap.test.tsx` was NOT created — no React DOM test infrastructure is installable under the npm lockdown (Vitest unavailable, no jsdom, no testing-library). Modal radio render is verified manually in Plan 06 UAT.
