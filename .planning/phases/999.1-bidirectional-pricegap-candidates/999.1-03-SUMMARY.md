# Phase 999.1 Plan 03: Direction i18n Keys Summary

Added 5 paired flat-dotted i18n keys under `pricegap.candidates.modal.direction.*` to both `web/src/i18n/en.ts` and `web/src/i18n/zh-TW.ts` in a single commit, backing the radio toggle that Plan 04 will render in the candidate modal.

## Implementation

Inserted the five new keys between the existing `modeledSlippageBps.label` and `save` keys in both locale files:

- `direction.label` — "Direction" / "方向"
- `direction.option.pinned` — "Pinned (configured side only)" / "單向（僅依設定的多空腳）"
- `direction.option.bidirectional` — "Bidirectional (fire on either sign)" / "雙向（任一方向達門檻即觸發）"
- `direction.help` — long-form English / Chinese explanation referencing `long_exch` / `short_exch`
- `direction.validation.invalid` — error message for invalid direction value

zh-TW values use fullwidth punctuation (：，「」) per Plan 10-02 convention. English values use ASCII punctuation.

## Verification

- `grep -c 'pricegap.candidates.modal.direction'` returns **5** for both files (lockstep parity).
- `cd web && ./node_modules/.bin/tsc --noEmit` exits **0** with zero errors. The `Record<TranslationKey, string>` typing on `zhTW` enforces structural lockstep at compile time — any drift between en.ts and zh-TW.ts would have failed this check.
- No `test:i18n` script exists in package.json yet; Plan 04 (per its frontmatter) is responsible for adding the native-node-test parity check. Compile-time check is sufficient gate for this plan.
- Existing `pricegap.candidates.modal.*` keys unchanged.
