# Phase 999.1 Plan 01: Bidirectional Pricegap — Backend Foundation Summary

Direction-aware detector branch + pre-call executor leg-role swap + position observability fields, delivered with full backward compat and a latent-bug fix for pinned-mode sign blindness.

## What Was Built

**Backend foundation for bidirectional pricegap candidates.** Six Go files modified, one created. Direction is opt-in per candidate via a new `Direction` field on `PriceGapCandidate` (default `"pinned"`). The detector now sign-filters pinned candidates (positive direction only) — closing a latent Phase-8 bug where `barRing.allExceed` used `math.Abs` and silently fired any sign, placing wrong-side trades on inverse spreads. Bidirectional candidates fire on either sign, and the executor performs a single pre-call leg-role rewrite to swap wire-side BUY/SELL when an inverse spread fires. Position observability gains `FiredDirection`, `CandidateLongExch`, `CandidateShortExch` so Phase 10's D-11 active-position guard can match by configured tuple even when wire-side roles diverge.

### 2. [Rule 3 — Blocking] Test helper spread math fix
- **Found during:** Task 2 RED phase
- **Issue:** The `pushBarsAtSpread` helper initially set `pxLong = mid + signedBps/1e4 * mid`, but with both legs offset symmetrically by `half` from mid the resulting spread is `2*half/mid * 1e4 = 2*signedBps` (off by 2x). RED phase showed +250 expected but +500 measured.
- **Fix:** Changed to `half := signedBps / 2.0 / 10_000.0 * mid`. Same correction applied to `TestDirection_BidirectionalRejectsMixedSigns` per-bar loop.
